# ADR-0001: 选择"先平台后 C 端"路线

- **状态**：Accepted
- **日期**：2026-06-08
- **决策者**：项目主理人
- **相关**：[docs/01-phases.md](../01-phases.md)、[docs/04-review-notes.md](../04-review-notes.md)

---

## 背景

平安健康需要建设企业级 Agentic 平台，承接多个业务线的 Agent 落地需求：

- **已识别业务线**：销售助理（B 端）、研发问答（B 端）、HR 问答（B 端）、医疗能力辅助（C 端高响应）等
- **团队规模**：1-2 工程师
- **基础设施**：公司已有 LLM 网关（OpenAI 兼容）、KG、ES、DB
- **时间盒**：无严格约束

矛盾点：

1. 平台建设本身需要 6 人月起的工作量
2. 医疗 C 端 Agent 单独做也需要 3 人月起（首 token < 1s、千级并发、强合规）
3. 1-2 工程师同时做这两件事，必然两头都做不好

---

## 决策

**采用路线 A：先建设平台 + 内部 B 端 Agent（Phase 0-3），完成后再扩展 C 端 Agent（Phase 4）。**

具体节奏：

- Phase 0（4-5 周）：POC 框架，不绑业务
- Phase 1（4-5 周）：销售助理"产品问答 + 找匹配产品"两个用例
- Phase 2（6-8 周）：多 Agent / Skill / Subagent / 长任务 / 审计
- Phase 3（8-12 周）：规模化与生产化
- Phase 4（独立立项）：C 端医疗 Agent

**Phase 4 启动的前置条件**：Phase 0-3 全部完成，且平台稳定运行至少 6 周。

---

## 备选方案

### 路线 B：先做 C 端医疗、后做平台

**做法**：先把医疗问答做出来上线，不做平台；等业务跑起来再回头抽平台。

**为什么不选**：
- 业务驱动看似 ROI 高，但**第二个 Agent 还是要从零写代码**——平台永远做不出来
- C 端医疗的合规、降级、监控会把 1-2 个人耗光，没有余力抽象
- 医疗 C 端一旦出问题（错误诊断 / 合规违规），整个 Agentic 方向会被叫停，影响远超延期

### 路线 C：平台 + C 端并行（两个项目）

**做法**：1 人做平台底座，1 人做销售助理（在底座上跑），新招人做医疗 C 端 Agent。

**为什么不选**：
- 至少需要 3-4 个工程师才能起步
- 当前团队规模（1-2 人）不支持

**何时重启路线 C**：团队扩充到 ≥ 4 工程师，且平台 Phase 1 已完成。

---

## 后果

### 好处

- 1-2 工程师可承担，工作量匹配
- B 端 Agent 容错高、并发低，让平台慢慢迭代成熟
- 平台经过销售 / HR / 研发等多业务线验证后，C 端 Agent 是"在平台上配置一下"而非"从零造"
- 医疗 C 端的合规风险被推迟到平台稳定后，降低早期翻车概率

### 代价

- C 端医疗 Agent 大约延迟 9-12 个月
- 若业务方对 C 端医疗有强 KPI，需要单独沟通预期管理
- 销售助理 Phase 1 只跑 2 个用例（产品问答 + 找匹配产品），完整 6 用例要等 Phase 2

### 风险

- **风险 1**：业务方不接受 C 端延迟 → 缓解：用 Phase 1-2 的内部 Agent 持续输出价值，证明平台
- **风险 2**：内部 B 端 Agent 需求不足，平台缺乏真实迭代场景 → 缓解：Phase 1 强约束接销售助理这种"已有真实需求"的场景
- **风险 3**：路线选定后业务突然要求 C 端优先 → 触发本 ADR 重评（见下）

---

## 验证 / 退出

### 验证决策生效

每个 Phase 退出时检查：

- Phase 0：POC 框架可跑通 demo Agent（见 03-validation.md V0.x）
- Phase 1：销售助理 V1.4.1 / V1.4.4 / V1.4.5 通过，**且引擎层 grep "sales" 零命中**（V1.5）
- Phase 2：3 个 Agent 同时跑，新增 Agent 只改 YAML
- Phase 3：通过公司安全 / 合规评审

### 重新评估触发点

出现以下任一情况，重开本 ADR 评审：

1. 业务方要求 C 端医疗 6 个月内上线
2. 团队扩充至 ≥ 4 工程师（可考虑路线 C）
3. Phase 0 完成时间超过 8 周（说明节奏估错）
4. Phase 1 平台抽象验证（V1.5）失败（说明路线 A 的核心假设有问题）
